Le contexte

Outlook Calendrier, c'est insuffisant pour la réception

"On ne sait jamais les heures d'arrivée des camions."
C'est la réalité quotidienne de la réception. Un calendrier Outlook partagé n'est pas fait pour gérer des flux de livraison avec des règles métier complexes. Il faut un outil dédié.
📅 Situation actuelle — Outlook Calendrier
CE QUE VOUS VIVEZ AUJOURD'HUI
  • ✗Outlook partagé — n'importe qui peut modifier, supprimer ou écraser un RDV sans trace
  • ✗Pas de créneau horaire imposé — le camion arrive "dans la journée", personne ne sait quand
  • ✗5 camions peuvent arriver en même temps — embouteillage, perte de temps, stress
  • ✗Incompatibilités matières jamais vérifiées — un Soja Bio après un Soja Conv, personne ne le voit
  • ✗Camion non arrivé = appels téléphoniques en chaîne, aucune trace, report improvisé
  • ✗Aucun historique fiable — impossible de savoir qui a livré quoi, quand, et si c'était à l'heure
  • ✗La direction n'a aucune visibilité — pas de chiffres, pas de rapports, pas de pilotage
🚀 Avec le portail dédié
CE QUE ÇA CHANGE
  • ✓Créneau de 2h imposé — le camion sait qu'il doit être là entre 8h et 10h, pas "dans la journée"
  • ✓Max 4 à 6 camions par créneau selon la matière — plus d'embouteillage, flux maîtrisé
  • ✓Incompatibilités vérifiées automatiquement — le transporteur ne peut pas choisir un créneau incompatible
  • ✓Accès mobile sécurisé — le chauffeur vérifie son RDV depuis son téléphone, depuis la route
  • ✓Non-arrivé signalé en 1 clic — report automatique sur le prochain créneau compatible
  • ✓Historique complet et inviolable — qui a livré quoi, quand, et si c'était à l'heure
  • ✓Rapports PDF pour la direction — ponctualité par transporteur, tonnage par matière

La gestion des incompatibilités matières

Certaines matières ne peuvent pas se succéder dans un silo sans risque de contamination croisée ou de perte de certification. Le portail gère tout automatiquement.

Comment ça marche : Le système regarde la dernière matière livrée avant un créneau. Si la matière demandée est incompatible avec la précédente, le créneau est automatiquement affiché comme indisponible pour le transporteur — sans explication technique. Si une matière neutre passe entre les deux, l'incompatibilité est levée.

Déploiement

Comment imposer les RDV à tous les transporteurs ?

La question clé : comment passer d'un système informel à un portail obligatoire, sans frictions ?

Le principe : On ne change pas les habitudes en demandant — on change les habitudes en imposant une contrainte simple. Si le portail est la seule façon de livrer, tout le monde l'utilise.
1

Créer un compte pour chaque transporteur

L'Admin crée les comptes depuis l'onglet "Utilisateurs". Email + mot de passe. Le transporteur reçoit ses identifiants. Simple comme un accès email.

💡 Vous pouvez créer autant de comptes "Transporteur" que nécessaire. Chaque transporteur peut aussi créer ses propres chauffeurs.
2

Communication officielle — "à partir du [date], RDV obligatoire en ligne"

Envoyez un email ou courrier à tous vos transporteurs avec le lien du portail et leurs identifiants. Précisez clairement qu'aucun camion ne sera accepté sans RDV confirmé sur le portail.

💡 Le portail est accessible depuis n'importe quel téléphone, tablette ou PC. Aucune installation requise.
3

Afficher le lien du portail à l'entrée du site

Un QR code à l'entrée qui pointe vers le portail. Un chauffeur qui arrive sans RDV peut en faire la demande immédiatement depuis son téléphone — s'il reste une place sur le créneau.

💡 L'URL est courte et mémorisable : reception-sanders.fr
4

Règle ferme : pas de RDV = pas de déchargement

L'employé à la réception vérifie le statut dans le portail avant d'accepter un camion. Si le RDV n'est pas dans le système, le camion est refusé ou doit attendre un créneau disponible.

💡 Le planning journalier imprimable peut être affiché en salle de réception pour vérification rapide.
5

Suivi et incitation par la ponctualité

Les rapports de ponctualité par transporteur créent une pression positive. Les meilleurs transporteurs peuvent être privilégiés pour les créneaux de choix. Les mauvais payeurs sont identifiés facilement.

💡 Le classement de ponctualité peut être communiqué mensuellement aux transporteurs pour les responsabiliser.
6

Formation en 5 minutes — le portail est simple

L'interface est pensée pour des chauffeurs non-techniciens. 3 étapes : se connecter → choisir une date → choisir un créneau. En cas de problème, l'Admin peut créer le RDV à la place du transporteur.

💡 La page d'accueil explique le fonctionnement. Les créneaux indisponibles s'affichent grisés avec la raison.
Architecture

Comment fonctionne le site techniquement ?

Une architecture moderne, simple et sans serveur à maintenir.

Frontend
HTML · CSS · JavaScript
Le site est une application web monopage (SPA). Tout le code tient en 3 fichiers : index.html, style.css, app.js. Aucun framework, aucune dépendance complexe. Chargement ultra-rapide.
3 fichiers · ~3000 lignes
Base de données & Auth
Supabase (PostgreSQL)
Supabase est une plateforme open-source basée sur PostgreSQL. Elle gère la base de données, l'authentification des utilisateurs, les autorisations par rôle et les APIs en temps réel. Plan gratuit suffisant pour ce volume.
PostgreSQL · Auth · API REST
Hébergement
GitHub Pages
Les fichiers statiques sont hébergés sur GitHub Pages — gratuit, fiable, HTTPS automatique. Pour un déploiement professionnel, migration possible vers OVH, Infomaniak ou tout hébergeur web standard en 10 minutes.
Gratuit · HTTPS · CDN mondial
Sécurité
Authentification JWT + RLS
Chaque utilisateur s'authentifie avec email + mot de passe. Les tokens JWT expirent automatiquement. Row Level Security (RLS) sur Supabase : chaque rôle ne voit que ses données autorisées, même si quelqu'un contourne l'interface.
JWT · Row Level Security
PWA
Progressive Web App
Le site peut être installé comme une application native sur iPhone et Android. Un Service Worker gère le cache pour une utilisation en conditions de réseau dégradé. Accessible depuis l'écran d'accueil.
iOS · Android · Service Worker
Temps réel
Polling 30 secondes
Le tableau de bord se rafraîchit automatiquement toutes les 30 secondes. Les données sont toujours à jour sans avoir à recharger la page. Système léger qui ne surcharge pas la base de données.
Auto-refresh · Sans rechargement

Questions fréquentes

Questions des décideurs

Que se passe-t-il si internet tombe en panne sur le site ?
Le planning journalier imprimable permet à l'équipe de réception de travailler avec une feuille papier en cas de panne réseau. La PWA garde également en cache les dernières données chargées. En conditions normales, une connexion 4G/5G ou même 3G suffit pour utiliser le portail.
Les données sont-elles sécurisées et confidentielles ?
Oui. Les données sont stockées sur les serveurs Supabase (infrastructure AWS, régions EU). Chaque utilisateur n'accède qu'à ses données autorisées grâce au Row Level Security. Les mots de passe sont hachés (bcrypt), jamais stockés en clair. Les communications sont chiffrées HTTPS.
Peut-on connecter ce portail à notre ERP existant ?
Oui. Supabase expose une API REST standard. Il est techniquement possible de synchroniser les données avec un ERP (SAP, Sage, etc.) via des webhooks ou des appels API planifiés. Cette intégration nécessiterait un développement supplémentaire selon le système cible.
Combien de transporteurs et d'utilisateurs peut-on gérer ?
Le plan gratuit Supabase supporte jusqu'à 50 000 utilisateurs actifs par mois et 500 Mo de base de données — largement suffisant pour ce volume. En cas de croissance importante, le plan Pro Supabase (25$/mois) lève toutes les limites. Le portail lui-même ne pose aucune limite technique.
Que se passe-t-il si un transporteur prend un RDV pour une matière incompatible ?
Impossible. Le système vérifie en temps réel les incompatibilités avant d'afficher les créneaux disponibles. Si la matière choisie est incompatible avec la dernière matière livrée sur un créneau donné, ce créneau n'apparaît tout simplement pas comme disponible pour le transporteur. Il ne peut pas forcer la main.
Comment migrer vers un hébergement professionnel plus tard ?
Migration triviale. Les 3 fichiers (index.html, style.css, app.js) se déposent sur n'importe quel hébergeur web en FTP. La base Supabase reste la même — aucune migration de données nécessaire. Possibilité d'utiliser un nom de domaine personnalisé (ex: reception.sanders-euralis.fr) avec un certificat SSL gratuit Let's Encrypt.
Peut-on ajouter d'autres sites Sanders / Euralis ?
Oui, deux approches : (1) Multi-sites dans la même application — ajouter un champ "site" sur les RDV et filtrer par site selon l'utilisateur. (2) Instances séparées — dupliquer le portail pour chaque site, chacun avec sa propre base Supabase. La deuxième approche est plus propre pour des volumes importants.
